# 一、文档引言

## 1.4 术语与定义

1.  Client（客户端）：共4个，本人电脑为其中1个，负责发起认证请求、与V服务器进行加密通信，展示UI界面；

2.  AS Server（认证服务器）：负责身份验证、发放Ticket票据与会话密钥，维护客户端公钥证书；

3.  V Server（应用服务器）：部署可展示业务应用，验证Client票据、消息签名，实现与Client的双向不可否认通信；

4.  RSA加密：纯手写实现，不引用现成加密库，用于Client与V服务器之间的消息认证、数字签名与验签；

5.  自定义公钥证书：绑定Client/V服务器ID与公钥，由AS服务器签名，确保身份与公钥的合法性；

6.  封包/拆包：定义统一的网络传输格式，解决数据传输中的粘包问题，保障数据完整性；

7.  双向不可否认：通过RSA签名与验签机制，确保发送方无法抵赖已发送的消息，接收方无法篡改接收的消息。

# 二、项目整体目标

1.  搭建完整的Kerberos认证体系，包含4个Client、1个AS认证服务器、1个V应用服务器，实现分布式部署；

2.  实现多Client并发登录不卡顿，采用多线程机制处理请求；

3.  纯手写RSA加密算法，完成自定义公钥证书设计，实现ID与公钥的绑定；

4.  完成WebUI设计，清晰展示明文、密文、密钥、票据、签名等信息；

5.  实现Client与V服务器之间的消息认证与双向不可否认，设计规范的封包/拆包机制；

6.  在V服务器上部署可展示的业务应用程序，支持功能演示与抓包验证；

7.  小组4人分工协作，按时完成开发、测试与调试，确保系统可正常运行，满足课设全部要求。

# 三、核心功能需求

## 3.1 整体架构需求（分布式设计）

1.  分布式部署：AS服务器与V服务器独立部署，使用不同端口（建议AS：5000端口，V：5001端口），可运行在不同设备上，也可在同一设备不同进程运行；

2.  4个Client独立运行，其中本人电脑作为Client1，其余3名成员分别负责Client2、Client3、Client4的开发与调试；

3.  通信方式：Client与AS、Client与V之间采用HTTP+JSON通信，结合自定义封包/拆包机制，确保数据传输安全、无粘包。

## 3.2 多线程并发需求

1.  AS服务器与V服务器均支持多线程，每个Client的请求独立分配线程处理，确保4个Client同时登录、并发访问时不卡顿、无响应延迟；

2.  线程管理：避免线程泄露，确保请求处理完成后线程正常释放，支持并发请求数量≥4，响应时间≤1秒。

## 3.3 Kerberos认证流程需求

1.  认证发起：Client向AS服务器发送身份ID，发起认证请求；

2.  票据发放：AS服务器验证Client身份合法性后，生成会话密钥（对称密钥）与Ticket票据，使用AS私钥对Ticket签名，将Ticket与会话密钥返回给Client；

3.  访问请求：Client携带Ticket、自身公钥证书、加密消息及签名，向V服务器发起访问请求；

4.  验证流程：V服务器依次验证Ticket有效性、Client公钥证书合法性、消息签名正确性，验证通过后允许Client访问业务应用；

5.  双向通信：Client与V服务器通信时，均需对消息进行加密与签名，接收方进行解密与验签，实现双向不可否认。

## 3.4 加密与认证需求

3.4.1 RSA加密实现（禁止引用现成加密库）

1.  手写核心模块：大整数运算（加法、减法、乘法、模幂、快速幂）、欧几里得算法求gcd、扩展欧几里得求逆元、米勒-拉宾素性检测；

2.  RSA功能：生成密钥对（公钥e、n，私钥d、n）、消息加密、消息解密、数字签名、签名验签；

3.  签名与验签：发送方使用自身私钥对消息哈希值签名，接收方使用发送方公钥验签，验签通过则确认消息未被篡改且发送方不可否认。

3.4.2 对称加密需求

1.  手写简易对称加密算法，用于Client与V服务器之间的消息加密传输；

2.  会话密钥：由AS服务器生成，用于Client与V服务器之间的对称加密，每次认证生成新的会话密钥，提升安全性；

3.  UI展示：WebUI中清晰展示明文、密文、会话密钥，便于演示与验证。

3.4.3 自定义公钥证书需求

1.  证书结构：包含所有者ID（Client/V服务器ID）、公钥（e、n）、有效期、签发者（AS服务器）、AS服务器签名；

2.  证书功能：实现ID与公钥的绑定，由AS服务器签名确认，V服务器通过验签确认证书合法性，避免公钥伪造；

3.  证书管理：AS服务器维护所有Client与V服务器的证书，支持证书有效性校验（有效期、签名验证）。

## 3.5 封包/拆包需求

1.  封包格式：采用“4字节长度前缀+JSON数据”的格式，长度前缀表示后续JSON数据的字节数，避免粘包；

2.  拆包流程：接收方先读取4字节长度前缀，再根据长度读取完整JSON数据，解析得到消息、签名、证书等内容；

3.  数据内容：封包数据需包含必要信息（Client ID、消息内容、密文、签名、证书、Ticket等），确保传输完整性与可解析性。

## 3.6 WebUI界面需求

1.  页面数量：至少包含4个核心页面，适配所有Client端（含本人电脑）；

2.  首页：选择Client身份（Client1~4），发起认证请求入口；

3.  认证页面：展示认证状态、Ticket票据信息、会话密钥，显示认证是否成功；

4.  加密通信页面：输入明文，展示加密后的密文，发送消息后展示接收方响应（明文、密文）、签名与验签结果；

5.  业务展示页面：访问V服务器上的业务应用，展示业务数据（如访问日志、用户状态、票据信息等）；

6.  界面要求：简洁清晰、操作便捷，核心信息（明文、密文、密钥、签名、验签结果）突出显示，支持实时刷新。

## 3.7 V服务器业务应用需求

1.  业务功能：部署可展示的简易业务应用，包含但不限于：Client登录状态展示、访问日志记录与展示、加密消息收发面板、票据与证书信息查询；

2.  展示要求：业务数据实时更新，支持多Client并发访问时的业务展示，界面与Client端UI风格统一；

3.  验证支持：支持抓包演示，可通过Wireshark抓取传输数据，展示密文、签名、票据等内容，验证传输安全性。

## 3.8 抓包演示需求

1.  抓包范围：可抓取Client与AS、Client与V之间的所有传输数据；

2.  展示内容：通过Wireshark过滤指定端口（5000、5001），清晰展示证书包、Ticket、密文、签名等数据，验证传输内容的安全性与完整性；

3.  演示说明：提供抓包操作步骤，确保小组成员均可完成抓包演示，向trae展示项目的安全性。

# 四、非功能需求

## 4.1 性能需求

1.  并发性能：4个Client同时登录、并发访问V服务器时，系统无卡顿、无崩溃，响应时间≤1秒；

2.  加密性能：RSA加密/解密、签名/验签响应时间≤500ms，对称加密/解密响应时间≤100ms；

3.  稳定性：系统连续运行30分钟无异常，无内存泄露、线程阻塞等问题。

## 4.2 安全性需求

1.  加密安全：RSA密钥对生成安全，私钥不泄露，加密消息不可破解；

2.  身份安全：Client身份需经AS服务器验证，未认证Client无法访问V服务器业务；

3.  数据安全：传输数据均经过加密与签名，防止篡改、伪造，实现双向不可否认；

4.  证书安全：自定义证书由AS服务器签名，无法伪造，验签失败则拒绝通信。

## 4.3 可操作性需求

1.  部署简单：提供清晰的部署步骤，小组成员可快速启动AS服务器、V服务器与Client端；

2.  操作便捷：WebUI界面操作简单，无需复杂步骤即可完成认证、消息发送、业务访问；

3.  调试便捷：系统输出清晰的日志信息，便于小组成员排查开发、测试中的问题。

## 4.4 兼容性需求

1.  浏览器兼容：WebUI支持Chrome、Edge等主流浏览器，适配不同屏幕尺寸；

2.  系统兼容：Client端（含本人电脑）、服务器端支持Windows系统，确保小组所有成员设备均可正常运行。
